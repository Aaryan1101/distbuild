1. Project Overview


This guide walks you through building a distributed build system for Java Ś conceptually similar to distcc for C/C++ Ś that replaces your slow local Gradle builds by spreading compilation tasks across multiple machines on your network.

When you run ./gradlew build, instead of your laptop compiling every module sequentially, a coordinator node (your laptop) parses the project dependency graph, splits independent modules into parallel tasks, dispatches them to worker nodes (teammates' machines, CI servers, or spare VMs), and assembles the final JARs once results are returned.

1.1 What You're Building
ComponentWhat it DoesCoordinator (Master)Runs on your laptop. Parses Gradle files, builds the DAG, schedules tasks, collects results.Worker AgentRuns on any spare machine. Waits for tasks, compiles Java sources, returns .class files.Build CacheRedis-backed. Hashes inputs Ś if unchanged, skips recompilation entirely.gRPC ProtocolFast binary transport between coordinator and workers. Typed, versioned, language-agnostic.Gradle PluginHooks into ./gradlew build so devs don't need to change their workflow.DAG SchedulerTopological sort + parallel task queue. Decides what runs now vs. what must wait.
1.2 Expected Speedup
ScenarioExpected ResultClean build, 3 worker nodes3¢4x faster compilationIncremental build (nothing changed)Near-instant Ś served from cacheIncremental build (1 module changed)Only that module recompiles, rest from cache10 independent modules, 5 workersUp to 5x parallelism on compilation phase


2. System Architecture


The system follows a coordinator-worker pattern with a shared artifact cache. All communication happens over gRPC on your local network. Workers are stateless Ś they receive everything they need in each task payload.

2.1 Data Flow (Step by Step)
1. Developer runs ./gradlew build on laptop
2. Gradle plugin intercepts the build lifecycle
3. Coordinator calls Gradle Tooling API to extract the project model (all modules + dependencies)
4. DAG Scheduler performs topological sort Ś identifies modules with no unresolved dependencies
5. For each ready module: SHA-256 hash of (source files + classpath JARs + compiler flags) is computed
6. Cache hit? Return .class files immediately Ś no compilation
7. Cache miss? Dispatch CompileTask to an available worker via gRPC
8. Worker receives task, runs javac or ECJ, returns compiled output
9. Coordinator stores result in cache, marks module complete
10. Newly unblocked modules are scheduled. Repeat until all modules done.
11. Coordinator assembles final JAR on your laptop. Build complete.

2.2 Component Interaction
FromTo / ActionGradle PluginCalls Coordinator API with project root pathCoordinatorCalls Gradle Tooling API (in-process) to get project modelCoordinator ? CacheGET hash ? returns compiled artifact or MISSCoordinator ? WorkergRPC CompileTask RPC with sources + classpathWorker ? CoordinatorgRPC CompileResult with .class files (zipped)Coordinator ? CachePUT compiled artifact with input hash as keyCoordinator ? GradleReturns assembled JARs to continue build lifecycle

4.2 proto/ Ś Shared Protocol
Protobuf files define the contract between coordinator and workers. Generate Java classes from these.

proto/
??? src/main/proto/
    ??? build_service.proto        # Main gRPC service definition
    ??? compile_task.proto         # CompileTask message (sources, classpath, options)
    ??? compile_result.proto       # CompileResult message (.class files, errors, timing)

FilePurposebuild_service.protoDefines the BuildService gRPC service with SubmitTask, GetTaskResult, and Heartbeat RPCscompile_task.protoCompileTask message: list of source files (bytes), list of JAR hashes, compiler flags, module name, input hashcompile_result.protoCompileResult message: compiled .class files (zipped bytes), stdout/stderr, success flag, duration_ms
4.3 coordinator/ Ś Master Node
coordinator/
??? build.gradle.kts
??? src/main/java/com/distbuild/coordinator/
    ??? CoordinatorMain.java          # Entry point Ś starts gRPC server, REST API
    ??? BuildCoordinator.java         # Orchestrates full build: DAG ? schedule ? collect
    ??? TaskScheduler.java            # Pulls ready tasks from DAG, dispatches to workers
    ??? WorkerRegistry.java           # Tracks available workers (IP, port, load, health)
    ??? WorkerClient.java             # gRPC stub for calling worker nodes
    ??? GradleProjectParser.java      # Calls Gradle Tooling API, extracts module graph
    ??? InputHasher.java              # SHA-256 of sources + classpath + flags
    ??? ResultCollector.java          # Receives worker results, writes to output dir
    ??? BuildSession.java             # Per-build state: task queue, results, status

FileWhat it accomplishesCoordinatorMain.javaStarts two servers: a gRPC server (port 9090) for worker callbacks, and an HTTP server (port 8080) for the Gradle plugin to call. Loads config (worker addresses, cache URL).BuildCoordinator.javaThe main orchestrator. Given a project root, calls GradleProjectParser to get the module graph, hands it to the DAG engine, then enters the scheduling loop: poll for ready tasks, dispatch, await results, unlock dependents.TaskScheduler.javaMaintains a priority queue of ready tasks. Assigns tasks to workers using a simple least-loaded strategy. Handles timeouts Ś if a worker goes silent, re-queues the task to another worker.WorkerRegistry.javaKeeps a live map of worker nodes. Workers heartbeat every 5 seconds. Registry marks workers as unhealthy after 3 missed heartbeats. TaskScheduler only pulls from healthy workers.WorkerClient.javaWraps the gRPC stub for BuildService. Handles channel lifecycle, deadlines, and retries with exponential backoff (3 attempts max).GradleProjectParser.javaUses Gradle Tooling API (GradleConnector) to extract a rich project model without running a full build. Returns a list of modules with their source dirs, compile classpath, and dependency edges.InputHasher.javaComputes a deterministic SHA-256 hash for each compile task. Input: sorted list of source file SHA-256s + sorted list of classpath JAR hashes + compiler flags string. Same inputs ? same hash ? cache hit.ResultCollector.javaReceives CompileResult from workers. Extracts .class files to the module's build/classes directory. Stores the result in cache. Signals BuildSession that this module is complete.BuildSession.javaImmutable-ish state for one build invocation. Tracks: total modules, completed modules, failed modules, start time, per-module timings. Used for progress reporting and final summary.
4.4 worker/ Ś Worker Agent
worker/
??? build.gradle.kts
??? src/main/java/com/distbuild/worker/
    ??? WorkerMain.java               # Entry point Ś registers with coordinator, starts gRPC
    ??? WorkerService.java            # gRPC service impl Ś receives tasks, returns results
    ??? JavaCompiler.java             # Wraps ECJ compiler Ś compiles source files in-memory
    ??? ClasspathResolver.java        # Resolves JAR hashes to actual files from local cache
    ??? SandboxManager.java           # Creates/cleans temp dirs per task for isolation
    ??? WorkerConfig.java             # Config: coordinator address, cache URL, thread pool size

FileWhat it accomplishesWorkerMain.javaStarts the worker gRPC server on a configurable port. Sends a RegisterWorker RPC to the coordinator at startup. Starts the heartbeat scheduler (pings coordinator every 5s). Handles SIGTERM gracefully Ś drains in-flight tasks before shutting down.WorkerService.javaImplements the BuildService gRPC interface. Receives SubmitTask RPCs, checks local JAR cache, delegates to JavaCompiler, serializes results, returns CompileResult. Uses a thread pool (default: CPU cores - 1) to handle concurrent tasks.JavaCompiler.javaWraps Eclipse ECJ in embedded mode (no subprocess). Accepts source file bytes + classpath JARs, runs compilation in-memory, captures diagnostics (errors, warnings with file/line), returns .class file bytes. Much faster than forking javac.ClasspathResolver.javaWorkers need the actual JAR bytes to compile. InputHasher sent JAR hashes Ś this class checks local disk cache first, then fetches from the shared Redis cache. Caches JARs locally so the next task that needs the same JAR doesn't re-download.SandboxManager.javaEach task gets a fresh temp directory under /tmp/distbuild/{taskId}/. Source files are written here before compilation. Directory is deleted after task completes (success or failure). Prevents cross-task interference.WorkerConfig.javaReads config from environment variables or a YAML file: coordinator host/port, Redis URL, max concurrent tasks, local JAR cache path, worker ID (defaults to hostname).
4.5 cache/ Ś Build Cache
cache/
??? build.gradle.kts
??? src/main/java/com/distbuild/cache/
    ??? BuildCache.java               # Interface: get(hash), put(hash, artifact), invalidate
    ??? RedisBuildCache.java          # Redis implementation using Lettuce async client
    ??? LocalDiskCache.java           # Fallback: file-system cache for single-machine mode
    ??? CacheKey.java                 # Value object wrapping the SHA-256 input hash
    ??? CacheEntry.java               # Cached artifact: .class bytes + metadata (timestamp, size)

FileWhat it accomplishesBuildCache.javaSimple interface with three methods: get(CacheKey) ? Optional<CacheEntry>, put(CacheKey, CacheEntry), invalidate(CacheKey). Both coordinator and workers depend on this interface Ś swap Redis for anything else without touching other code.RedisBuildCache.javaLettuce-based async Redis client. Serializes CacheEntry to MessagePack (compact binary). Uses Redis TTL (default 7 days). Handles connection pooling and reconnection. Key format: distbuild:v1:{sha256hex}.LocalDiskCache.javaFile-system implementation for teams without Redis. Stores entries as files under ~/.distbuild/cache/{first2hex}/{sha256hex}. Implements LRU eviction when cache exceeds configured size (default 10 GB). Good for getting started without infrastructure.CacheKey.javaImmutable value object. Wraps the 64-char hex SHA-256 string. Has equals/hashCode. Validates hex format on construction. Used as the canonical identifier for any compile task's output.CacheEntry.javaContains: byte[] classFiles (zipped), String moduleId, long createdAt, long sizeBytes, Map<String,String> metadata (compiler version, Java version). Metadata enables cache invalidation when compiler is upgraded.
4.6 dag/ Ś Dependency Graph Engine
dag/
??? build.gradle.kts
??? src/main/java/com/distbuild/dag/
    ??? ModuleGraph.java               # The DAG: nodes = modules, edges = dependencies
    ??? Module.java                    # Node: id, sourceDirs, classpath, dependsOn
    ??? TopologicalSorter.java         # Kahn's algorithm Ś returns layered build order
    ??? ParallelTaskQueue.java         # Thread-safe queue of currently-runnable modules
    ??? CycleDetector.java             # Detects circular dependencies, throws with clear message

FileWhat it accomplishesModuleGraph.javaAdjacency list graph of Module nodes. Supports: addModule(), addDependency(), getRoots() (modules with no dependencies), getSuccessors(module) (what becomes available when this module completes). Immutable after construction.Module.javaValue object for one Gradle subproject: moduleId (e.g. :core, :api), list of source directories, list of classpath JAR paths, list of dependency module IDs. Built by GradleProjectParser.TopologicalSorter.javaKahn's algorithm on the DAG. Returns not a flat list but a list of layers Ś layer 0 is all roots (can start immediately), layer 1 becomes available once layer 0 completes, etc. Layers let you visualize max parallelism per phase.ParallelTaskQueue.javaThread-safe blocking queue. Coordinator polls it for ready tasks. When a module completes, its successors are checked Ś if all their dependencies are done, they're added to the queue. Central to the scheduling loop.CycleDetector.javaDFS-based cycle detection on the module graph. Runs before any scheduling starts. If a cycle is found, throws CyclicDependencyException listing the full cycle path. Rare in real projects but critical to catch early.

7. Hard Problems and How to Solve Them


7.1 Classpath Correctness
Workers must compile with the exact same JARs that a local build would use. Sending JARs over the network per task is wasteful. Solution: content-addressed JAR cache.

12. Coordinator computes SHA-256 of every JAR on the local classpath
13. CompileTask message sends only the list of hashes Ś not the JAR bytes
14. Worker checks its local disk cache for each hash
15. Missing JARs are fetched from Redis (coordinator uploaded them there at build start)
16. JARs are stored locally so the next task that needs them skips the download

7.2 Annotation Processors
Annotation processors (Lombok, MapStruct, Dagger) run at compile time, generate source files, and are often stateful. They are incompatible with distributed compilation in the general case.

Strategy for Phase 1: Detect modules that declare annotation processors in their annotationProcessor configuration. Fall back to local compilation for those modules. Distribute everything else. This handles 80% of real projects correctly.

Strategy for Phase 2 (advanced): Run annotation processing locally first, then distribute the actual compilation of the generated sources. Requires splitting the JavaCompile task into two phases.

7.3 Incremental Compilation State
Gradle's incremental compiler tracks which .class files depend on which source files. This state lives in build/tmp/compileJava/. Workers don't have this state.

Solution: Include the incremental state directory in the cache key. If the state is missing (cold worker), do a full compile of that module. Full compiles are still fast for individual modules. The coordinator's cache hit means most modules never reach workers anyway.

7.4 Worker Discovery on a LAN
For a team of 10, manual IP configuration (workers listed in coordinator config) is fine and avoids DNS/mDNS complexity. For more dynamic setups, use mDNS: workers broadcast _distbuild._tcp.local at startup, coordinator listens and auto-discovers them.

7.5 Security
For an internal team on a trusted LAN, no auth is needed in Phase 1. For later: add mutual TLS to gRPC channels (gRPC has excellent mTLS support). This prevents rogue machines from joining your worker pool.

9. Common Pitfalls to Avoid


Ģ Use content-addressed hashes. Workers fetch JARs they don't have from cache. Sending 50 MB of JARs per task over the LAN will be slower than local compilation.Don't send full JAR bytes in CompileTask: 

Ģ BuildSession should be immutable once created. Use ConcurrentHashMap for tracking completion. Race conditions in the scheduler cause mysterious build failures.Don't share mutable state between tasks: 

Ģ If the coordinator is unreachable, the Gradle plugin MUST fall back to local compilation silently. Developers will not tolerate a build tool that breaks their build.Don't skip the fallback path: 

Ģ All workers must run the same JDK version as your coordinator. Compiling with Java 21 on the worker and running on Java 17 locally causes cryptic errors. Embed the target Java version in the cache key.Don't forget Java version alignment: 

Ģ Only store CompileResult in cache if success == true. A cached failure would mean a fixed source file still fails on next build.Don't cache failed compilations: 

Ģ Paths differ across machines (/Users/alice vs /Users/bob). Hash only file contents, not paths. This is why InputHasher reads file bytes, not file names.Don't use file paths in cache keys:
